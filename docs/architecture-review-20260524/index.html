Problem

Every form screen reimplements the same three dicts, live-validation listeners, error-panel toggle, and a fragile _SelectCurrent private import.

Solution

A FieldValidator module whose interface is set_error / clear / validate_input / validate_select — the _SelectCurrent walk lives inside it, invisible to callers.

Wins

  • ✦ locality: Select styling bug fixed in one place
  • ✦ leverage: one interface, 3+ form screens
  • ✦ private import isolated behind the seam
  • ✦ interface is the test surface for all form validation

  • ✦ locality: off-by-one fixed in one place
  • ✦ Paginator is trivially unit-testable without a Textual app

5 — Introduce a FeatureScreen base class

Every features/*/screen.py — 41 files each declare:
CSS_PATH = Path(__file__).parent / "styles.tcss"  +  Binding("escape", "action_go_back", "Back")  +  def action_go_back(self): self.app.pop_screen()

Before — 41 × identical boilerplate

Each of 41 screen.py files:
CSS_PATH = Path(__file__).parent / "styles.tcss"
Binding("escape", "go_back", "Back")
def action_go_back(self): self.app.pop_screen()
… actual screen logic …
× 41 screens = 123 redundant lines

After — inherited, invisible

tui/base.py
class FeatureScreen(Screen[None]):
  CSS_PATH = auto-resolved
  BINDINGS = [escape → go_back]
  action_go_back() → pop_screen()
Each screen:
class MyScreen(FeatureScreen): …only logic…

Problem

41 screens each declare the same CSS_PATH, Escape binding, and action_go_back. A screen that forgets any of them breaks silently.

Solution

A FeatureScreen base that auto-resolves CSS_PATH from the subclass's __module__ and provides the back-navigation binding by default.

Wins

  • ✦ leverage: 3 lines enforced for all 41 screens
  • ✦ new screens are correct by construction
  • ✦ deletes ~123 lines of boilerplate
Note: Auto-resolving CSS_PATH from __module__ requires that the package layout stays stable. Screens that want a different CSS file would need to override the attribute explicitly — acceptable tradeoff.
Speculative locality

6 — Unify screen navigation through navigate()

tui/router.py  ·  features/search_grid/screen.py (line 652)  ·  features/dashboard/screen.py (line 72)

Before — two navigation patterns

# gallery → any demo screen
navigate(self.app, "searchgrid")
# search_grid → edit screen
self.app.push_screen(EditJobScreen(record_id=key))
# dashboard → detail screen
self.app.push_screen(DetailScreen(item_id=…))

After — one pattern everywhere

# gallery → any demo screen
navigate(self.app, "searchgrid")
# search_grid → edit screen
navigate(self.app, "editjob", record_id=key)
# dashboard → detail screen
navigate(self.app, "detail", item_id=…)

Problem

Child screens (edit, detail) are pushed directly via app.push_screen() while sibling screens use navigate(). The seam exists in the router but not all callers cross it.

Solution

Register child screens in routes.py with their constructor kwargs as the contract. All navigation goes through navigate(); app.push_screen() is not called in feature code.

Wins

  • ✦ locality: all navigation logic in router.py
  • ✦ one pattern to learn for new screen authors
  • ⚠ only one adapter today — seam is hypothetical

Top Recommendation

Start with #2 + #3 together

Centralizing color maps (#2) and making _data.py the single source of truth for search_grid (#3) are both one-hour changes with zero risk and immediate locality gains. They remove dead code, make existing docstrings true, and unblock the larger refactors that follow.

Then tackle #1 (FieldValidator) — the highest-leverage deepening. The _SelectCurrent workaround is the sharpest thorn: a private Textual import that bleeds into three screen files. Pulling it behind a seam concentrates both the fragility and any future fix.
